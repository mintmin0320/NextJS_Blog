# 프로젝트 현황
오늘도 어김없이 내일 발표를 위한 프로젝트를 진행 중이다. 어제 있었던 일정 등록 파트는 모두 끝냈고 나머지 1차 데모 발표 때 발표할 공유 오피스와 회의록 관련 파트를 진행한 내용을 간단하게 정리하려고 한다.

<br/>

## 회의록 페이지
간단하게 설명하면 react-speech-recognition 라이브러리를 사용해 사용자의 음성 녹음이 가능한데, 이 기능을 이용해 회의를 할 때 따로 적을 필요 없이 녹음 시작 버튼을 누르고 회의가 끝났을 때 저장 버튼을 클릭하면 해당 텍스트 정보를 DB에 저장하고 관리할 수 있도록 설계했다. 음성인식에 대한 큰 기대를 하지 않았지만 생각보다 인식률이 높았다. 오늘 나를 고민되게 했던 부분은 상태 관리와 연관되어 있는 부분이다.

현재 Minutes 컴포넌트에 MinutesList(회의록 리스트 컴포넌트)와 DetailMinutes(희의록 상세보기) 컴포넌트가 속해있는 상황에서 좌측 회의록 리스트에 상세보기 버튼을 클릭하면 우측 상세보기 페이지에 상세 정보가 표시되는 로직이다.
![](https://velog.velcdn.com/images/mintmin0320/post/8ff35f10-31be-4c91-8788-b632f81e5b28/image.png)

만약 두 컴포넌트가 같은 위치에 존재하거나 Minutes 컴포넌트가 부모 컴포넌트에 위치했다면 손쉽게 해결할 수 있었겠지만 props로 관리하기에는 한계가 있다고 생각해 ***useContext***를 사용했다.
여기서도 두 가지 방법에서 고민을 했었는데, 회의록 id만 넘겨서 서버에 해당 id에 해당하는 정보를 요청하거나 리스트 값을 모두 넘겨서 출력하는 방법이 있다고 생각했다.
서버에 요청을 보내는 소요를 줄이는 편이 좋다고 생각해서 버튼 클릭 시 해당 데이터를 넘기는 방향으로 노선을 정했다. 다행히도 비슷한 상황을 겪었던 경험이 있었기 때문에 시간은 조금 소요됐지만 구현은 할 수 있었다.

<br/>

### 상태 관리
구현 방법도 고민했지만 가장 고민이 되었던 부분은 어떤 상태 관리 라이브러리나 hook을 적용할지에 관한 고민이었다. 
1. Redux
2. useContex
3. React Query
일단 개인적인 생각으로는 Redux를 적용할 필요까지는 없다고 생각했다. 서버 데이터를 Redux로 관리하기보다는 React Query를 적용해 보려고 했으나 아직 숙련도가 부족해 일단은 useContext를 사용했다. 사실 예전부터 React Query를 사용해 보고 싶은 마음은 강했지만 시간에 쫓기는 상황에서 미숙한 기술을 적용하기에는 무리가 있다고 생각했다. 일단 이번 발표가 끝나게 된다면 여유를 두고 코드를 개편할 계획이다!

# 프리보온딩
바쁜 와중에 오늘 프리보온딩 2주차가 진행된다는 사실을 뒤늦게 알게 되었다. 1회차 때 듣지 못해서 이번에는 놓치지 않겠다는 다짐을 했었기 때문에 저녁을 먹으며, 쉬는 시간에 잠시 들을 수 있게 되었다.
짧았지만 강사님께서 말씀 중 나에게 충격으로 다가왔던 내용이 있다.
1. "버튼을 div로 만들고 pointer 사용해서 버튼처럼 사용하지 마세요.."
2. "모든 태그를 div로 만드시는 분, 시맨틱 태그 잘 안 쓰는 분들 많으시죠? 그러지 마세요"

사실 두 가지 모두 나에게 해당되는 말이다.. 어느 순간부터 버튼을 div로 만드는 게 습관이 되어 자주 사용했었고 시맨틱 태그는 나중에 정리하겠다는 마음으로 모두 div를 사용했었다.
말씀해 주신 내용을 듣고 그동안 너무 어려운 내용, 아직 들을 부분이 아닌 내용만 알고 싶은 마음에 기본이 되는 부분을 놓친 것 같다. 나름 클린 코드를 지향하며 열심히 작성해왔다고 생각했지만 실상은 시맨틱 태그나 버튼 같은 기본도 하지 못하는 상황이었다. 발표가 끝나게 된다면 HTML 태그에 대한 기본부터 다시 생각하고 정리할 필요가 있을 것 같다.
